# Stage 4: High-Complexity Environment Scaling and Mutation Metric Refinement

The Stage 3 long-run experiments (3000 episodes) successfully demonstrated stable training for both I-MAPPO and the MAPPO baseline. However, for a top-tier academic paper, the results show two issues:
1. **The gap between I-MAPPO and MAPPO is too small at the end of 3000 episodes.** The environment is likely too simple (4 UAVs), allowing the standard MAPPO to catch up eventually. We need to demonstrate that I-MAPPO drastically outperforms MAPPO in **high-complexity/high-density** scenarios.
2. **The Zero-Shot Intent Mutation test occasionally returns `null`.** The strict condition ("all UAVs must completely reverse velocity") is too rigid, causing a failure if even one UAV performs a brief obstacle-avoidance maneuver.

Please update the `uav_scheduling_env.py`, `imappo_experiments.py`, and `test_intent_mutation.py` codebase to execute the following 2 tasks:

## Task 1: Scale Up Environment Complexity (The "8-UAV Swarm" Scenario)
To prove the superiority of the Cross-Attention Critic and Action Masking, we need to stress-test the algorithms in a much more complex state-action space.
Please update the default arguments in `imappo_experiments.py`:
- Increase the number of agents to `n_agents = 8`.
- Increase the number of targets to `n_targets = 6`.
- Keep the episodes at `1500` (since we mainly want to show that I-MAPPO converges much faster in the early/mid stages, whereas MAPPO will likely struggle to converge quickly in this 8-agent setting).
- Ensure the observation and state dimensions in the environment automatically scale or are updated to match `N=8` and `M=6`.

## Task 2: Refine the Intent Mutation Metric (Eliminate `null` Latency)
We need a more robust and academically acceptable way to measure "Response Latency" in `test_intent_mutation.py`.
Please rewrite the metric calculation logic in `test_intent_mutation.py` with the following relaxed criteria:
1. **Increase Evaluation Horizon:** Set `--total-steps 100` and trigger the intent mutation at `step 30` (giving the swarm more time to react).
2. **Centroid-Based Evaluation:** Instead of requiring every single UAV to reverse its velocity vector immediately, measure the **Average Distance** from the swarm to the targets.
   - *Logic:* Before step 30 (Gather intent), the average distance between the UAVs and their targets should be decreasing.
   - After step 30 (Evade intent), track the average distance at each step. 
   - **Response Latency** is defined as: `Current Step - 30`, where the "Current Step" is the *first* step after mutation where the **Average Distance of the swarm to the targets strictly increases for 3 consecutive steps** (to filter out noise).
3. Ensure the script logs this new latency metric and saves the trajectory `(x, y)` over the 100 steps so we can plot a clear "U-turn" trajectory map for the paper.
